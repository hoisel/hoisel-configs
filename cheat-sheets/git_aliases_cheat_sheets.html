Cheat Sheet — Seus atalhos de Git

Um guia rápido (com exemplos) para os aliases personalizados do seu ~/.gitconfig que você usa no dia a dia.

config branches status fetch stash tags worktree
Data: 03/03/2026
Formato: pt-BR
Escopo: aliases globais
Como usar
Todos os exemplos assumem que você está dentro de um repositório Git (exceto git ce).
git <alias> [args...]
Exemplo:
git bb feature/minha-feature
git ss
Dica rápida
  • switch (git b/git bb) é a forma moderna de trocar/criar branches.
  • checkout (git co/git cb) continua útil para restaurar arquivos.
  • Aliases com ! executam shell (funções), então aceitam parâmetros e podem fazer várias etapas.

Fluxos práticos (receitas rápidas)

Combinações comuns de aliases para tarefas reais do dia a dia.

Criar branch nova (moderno)
Cria e troca para a branch:
git bb feature/minha-feature
Depois verifique:
git ss
Trocar de branch
Troca para main (moderno):
git b main
Versão clássica:
git co main
Checkout para restaurar arquivo
Útil quando você quer “voltar” um arquivo:
git co -- README.md
Dica: isso descarta mudanças não commitadas no arquivo.
Atualizar tudo (todos remotes)
Somente atualizar refs:
git fa
Atualizar + limpar branches remotas apagadas:
git fap
Quando usar -p: depois de merges/deletes no servidor (ex.: PRs fechados).
Stash só do que está staged
Fluxo típico:
git add -p
git stash-staged
Você “guarda” o que já estava pronto (index), sem mexer no resto do working tree.
Remover tag local + remota
Remove a tag v1.2.3 (default remote: origin):
git rtag v1.2.3
Remover em outro remoto:
git rtag upstream v1.2.3
Esse alias tenta deletar localmente (se existir) e depois remove no remoto via push de ref vazia.
Gerar .gitignore na hora
Gera e salva em arquivo:
git ignore macos,windows,node > .gitignore
Dica: revise e comite o .gitignore no repositório.
Trabalhar em PR com worktree (GitHub)
Ideal para revisar PR sem bagunçar sua working tree atual.
# cria worktree em ../pr-123 e entra no diretório
git prw 123

# ...trabalhe/revise, rode testes, etc...

# envia seus commits de volta para a branch do PR
git pprw 123

# remove worktree e apaga a branch local pr-123
git prwrm 123
Observação: prw assume refs pull/<num>/head (padrão GitHub). pprw requer gh CLI.
Sugestão: use worktrees para reviews e hotfixes sem “poluir” sua árvore principal.

Notas técnicas (o que os aliases “shell” realmente fazem)

Esses atalhos usam ! no gitconfig: o Git executa uma função no shell. Bom para automatizar fluxos completos.

git rtag

Objetivo: remover tag local e no remoto.

Assinaturas aceitas:
git rtag <tag>
git rtag <remote> <tag>
Comportamento importante:
  • Se você não passar remote, usa origin.
  • Tenta deletar a tag local (ignora erro se ela não existe).
  • Remove no remoto via: git push <remote> ":refs/tags/<tag>"
Cuidado
Remover tags pode quebrar releases, pipelines e links de comparação. Use quando você tiver certeza.

git ignore

Objetivo: buscar templates de .gitignore por nome (API da Toptal/gitignore).

Exemplos comuns:
git ignore python,venv,visualstudiocode > .gitignore
git ignore node,macos,windows > .gitignore
Dica:
  • Você pode concatenar templates com vírgula.
  • Se quiser apenas visualizar, rode sem redirecionar.

git prw

Objetivo: criar um worktree para revisar/rodar testes de um PR sem trocar sua árvore atual.

Como funciona (alto nível):
  • Define branch=pr-<num> e dir=../pr-<num>
  • Faz fetch do PR e cria a branch local
  • Cria o worktree e entra no diretório
git prw 123
git prw 123 upstream
Quando falha:
  • Se o remoto não expõe refs pull/<num>/head (fora do padrão do GitHub).
  • Se já existe ../pr-<num> ou a branch pr-<num>.

git prwrm

Objetivo: remover o worktree e limpar a branch local do PR.

git prwrm 123
  • Remove ../pr-123 via git worktree remove
  • Apaga a branch local via git branch -D pr-123
Cuidado
Se você tiver commits locais não enviados na branch pr-123, eles serão perdidos. Use git pprw antes!

git pprw

Objetivo: fazer push dos seus commits locais para a branch remota de um PR (mesmo repo ou fork).

Assinatura:
git pprw <pr-number>
Como funciona (alto nível):
  • Usa gh pr view para descobrir a branch (headRefName) e o repositório (sshUrl) do PR.
  • Adiciona um remote temporário (prw-<num>) apontando para o repo do PR.
  • Faz git push do HEAD atual para a branch do PR.
  • Remove o remote temporário após o push.
Quando falha:
  • Se gh CLI não estiver instalado ou autenticado.
  • Se você não tiver permissão de push no repositório do PR (ex.: fork de terceiro).
Atenção
Se a branch atual não for pr-<num>, o alias avisa mas faz push do HEAD mesmo assim.
Checklist rápido de segurança
  • Antes de git rb: confirme se não tem commits locais importantes.
  • Antes de git rtag: confirme o nome exato da tag (ex.: git tg).
  • Antes de git prwrm: garanta que nada valioso ficou dentro do worktree do PR.
  • Antes de git pprw: confirme que o HEAD contém apenas as mudanças desejadas (ex.: git log --oneline).
Fim da cheat sheet — personalize à vontade e mantenha seus aliases documentados.
